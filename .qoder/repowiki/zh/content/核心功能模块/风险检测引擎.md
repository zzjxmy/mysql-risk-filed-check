## 简介
本文件面向"风险检测引擎"的技术文档，聚焦于后端核心类 CheckEngine 的实现与运行机制。该引擎负责扫描目标数据库中的数据库、表与字段，针对以下风险类型进行检测与评估：
- 整型溢出（含有符号/无符号）
- 小数溢出（DECIMAL/NUMERIC）
- Y2038 问题（TIMESTAMP）

同时，文档阐述了采样策略、阈值配置、白名单过滤、进度持久化与事务控制、异常处理与性能优化等关键技术点，并给出配置参数说明、调优建议与故障排除指引。**更新**：新增智能表扫描策略，包括空表跳过、主键发现、基于主键的采样检查等功能，显著提升大表处理性能。

## 核心组件
- 检测引擎（CheckEngine）：负责数据库扫描、表与字段枚举、风险检测、采样与阈值判断、结果落库与进度持久化。
- 连接服务（ConnectionService）：管理数据库连接配置、密码加密解密、连接测试与验证。
- 任务与执行（CheckTask、TaskExecution）：承载检测任务的配置与执行状态。
- 风险结果（RiskResult）：记录具体风险项、阈值、使用率、建议等。
- 白名单服务（WhitelistService）：按任务级别（全局/自定义/不使用）匹配数据库/表/字段白名单。
- 风险结果服务（RiskResultService）：查询、统计与 DTO 转换。
- 任务服务（TaskService）：任务 CRUD、关联告警配置、DTO 映射。
- 执行服务（ExecutionService）：日志推送、进度计算、WebSocket 实时监控。
- 应用配置（application.yml）：数据源、连接池、JPA 方言、日志、JWT、AES 密钥等。

## 详细组件分析

### CheckEngine：风险检测引擎核心
- **数据库连接优化**
  - JDBC URL配置包含连接超时（connectTimeout=30000ms）、套接字超时（socketTimeout=300000ms）、自动重连（autoReconnect=true）、最大重连次数（maxReconnects=3）。
  - 启用连接有效性验证（conn.isValid(5)），在处理大表前检查连接状态。
- 扫描范围与过滤
  - 获取数据库列表，支持模式匹配；排除系统库。
  - 枚举表清单，支持行数与模式过滤。
  - 枚举字段元数据（类型、精度、长度等）。
- 白名单过滤
  - 支持任务级白名单（全局/自定义/不使用），按数据库/表/字段粒度匹配。
- **智能表扫描策略**
  - **空表跳过**：自动跳过行数为0的表，避免无效扫描。
  - **主键发现**：自动检测表的主键列，用于后续采样策略。
  - **基于主键的采样检查**：对大表使用基于主键的最新数据采样，确保样本代表性。
  - **智能扫描决策**：根据表行数和任务配置自动选择采样或全量扫描策略。
- 风险检测算法
  - 整型溢出：基于列类型映射到最大/最小值，计算使用率，超过阈值即判定风险。
  - 小数溢出：根据精度与标度计算整数位上限，比较绝对值最大值与阈值。
  - Y2038：对 TIMESTAMP 列取最大时间，若年份达到阈值则告警。
- 采样与阈值
  - 对大表（行数大于阈值且未开启全量扫描）采用基于主键的随机抽样，抽样数量由任务配置决定。
  - 阈值百分比用于衡量使用率是否达到风险线。
- **增强的进度跟踪与日志监控**
  - 每处理一张表都更新进度并发送实时日志，格式为"[processed/total]"。
  - 智能扫描策略选择：根据表行数自动决定采样或全量扫描。
  - 格式化行数显示：支持亿、万级别的可读格式。
  - 批量保存优化：每处理5张表统一持久化一次，减少数据库写入压力。
- 中断与异常
  - 支持外部停止信号，及时退出。
  - 数据库异常统一捕获并抛出运行时异常。

### 整型溢出检测算法
- 数值上限映射：根据列类型（含 UNSIGNED）映射到对应的最大/最小值。
- 使用率计算：取实际最大/最小值，计算绝对值占上限的比例。
- 风险判定：当使用率超过阈值时，生成风险结果并附带建议。
- **智能采样**：对大表使用基于主键的最新数据采样，确保样本代表性。

### 小数溢出检测算法
- 上限计算：整数位 = 精度 - 标度，上限 = 10^(整数位) - 1。
- 实际比较：取绝对值最大值，计算使用率，超过阈值即风险。
- **智能采样**：对大表使用基于主键的最新数据采样，确保样本代表性。

### 采样策略与阈值配置
- **智能采样策略**：根据表行数和任务配置自动选择采样或全量扫描策略。
- **主键驱动采样**：对大表使用基于主键的最新数据采样，确保样本代表性。
- **空表跳过**：自动跳过行数为0的表，避免无效扫描。
- 大表采样：当表行数超过阈值且未开启全量扫描时，使用基于主键的 ORDER BY 主键 DESC LIMIT N 的方式采样。
- 阈值百分比：用于衡量使用率是否达到风险线。
- 配置项（来自任务实体）：
  - fullScan：是否强制全表扫描
  - sampleSize：抽样条数
  - maxTableRows：大表阈值
  - thresholdPct：阈值百分比
  - y2038WarningYear：Y2038告警年份
- 配置来源与默认值：
  - 任务创建/更新时设置默认值
  - 应用配置中包含连接池、JPA 等基础参数

## 性能考量
- **数据库连接优化**
  - 使用JDBC连接超时参数：connectTimeout=30000ms，socketTimeout=300000ms，有效防止长时间阻塞。
  - 启用自动重连机制：autoReconnect=true，maxReconnects=3，提高连接稳定性。
  - 连接有效性验证：conn.isValid(5)，在处理大表前检查连接状态。
- I/O 与网络
  - 使用连接池（HikariCP）降低连接开销，默认最大池大小与空闲超时等参数可在配置中调整。
- 扫描策略
  - **改进**：实现智能表扫描策略，包括空表跳过、主键发现、基于主键的采样检查，显著提升大表处理性能。
  - 空表跳过：自动跳过行数为0的表，避免无效扫描。
  - 主键驱动采样：对大表使用基于主键的最新数据采样，确保样本代表性。
  - 可以跳过大表（超过maxTableRows且未启用fullScan时），平衡准确性和性能。
- 写入与事务
  - 批量保存风险结果，定期持久化执行进度，减少数据库写入频率。
  - **新增**：批量保存优化，每处理5张表统一持久化一次。
- SQL 优化
  - 使用 information_schema 快速获取元数据；对大表采用基于主键的采样查询而非RAND()采样，避免索引与排序成本。
- 并发与中断
  - 支持外部停止信号，及时释放资源；应用配置中可限制并发任务数。
- **新增**：实时日志监控
  - WebSocket 实时推送执行状态，前端可实时查看进度和日志。
  - 日志格式化输出，包含详细的进度指示器和扫描策略信息。

## 故障排除指南
- **数据库连接失败**
  - 检查 JDBC URL、用户名与密码；确认解密密钥正确；核对连接池参数。
  - 验证连接超时参数配置是否合理（connectTimeout、socketTimeout）。
  - 使用ConnectionService.testConnection进行连接测试。
- **连接不稳定**
  - 检查自动重连配置（autoReconnect、maxReconnects）。
  - 监控连接有效性验证（conn.isValid(5)）的结果。
  - 调整连接超时参数以适应网络环境。
- 执行中断
  - 若中途停止，检查停止信号回调逻辑；确认任务状态为 STOPPED。
- 结果缺失
  - 检查批量保存与进度持久化的事务模板是否生效；确认阈值配置是否合理。
- 白名单误判
  - 校验规则格式（数据库.表.字段）、通配符转义与大小写不敏感匹配。
- 性能问题
  - 调整采样大小与大表阈值；必要时关闭全量扫描；监控连接池与数据库负载。
  - **新增**：对于超大表，考虑增加maxTableRows阈值或启用fullScan。
  - **新增**：检查主键发现功能是否正常工作，确保采样策略有效。
- **新增**：日志监控问题
  - 检查 WebSocket 连接状态；确认日志推送服务正常运行。
  - 验证日志格式解析是否正确，特别是"[processed/total]"格式。
  - 检查前端实时监控组件是否正确订阅 WebSocket 主题。

## 结论
该风险检测引擎以 CheckEngine 为核心，结合白名单过滤、智能采样策略与阈值配置，实现了对整型溢出、小数溢出与 Y2038 问题的自动化检测。**更新**：通过实现智能表扫描策略（空表跳过、主键发现、基于主键的采样检查），显著提升了大表处理性能和检测效率。新的智能扫描策略不仅提高了系统性能，还确保了检测结果的准确性。通过数据库连接优化（JDBC超时参数、自动重连机制、连接有效性验证）、增强的日志监控系统、格式化行数显示和批量保存优化，显著提升了用户体验和系统性能。通过合理的 I/O 与事务控制、连接池配置以及可调的采样与阈值参数，能够在保证准确性的同时兼顾性能与可运维性。建议在生产环境中结合业务规模与数据特征，动态调整采样与阈值，并建立完善的白名单与告警机制。实时日志监控功能使得用户能够全程跟踪检测进度，提高了系统的透明度和可控性。
